# 给另一个 AI 的 Skill 改造指令

你要修改的是飞书画板相关 Skill，使其在“需要图形与连线真实吸附、拖动节点时连线跟随、后续仍可继续编辑”的场景下，**强制走原生 raw / 原生 connector 路线**，不要走任何会把关系降级成几何线条或不可验证图元的路线。

## 目标

当前问题不是“线看起来没连上”，而是“线没有在结构上绑定到节点锚点”，导致拖动节点后，线不会跟着移动。你需要从 Skill 层面彻底约束这种错误。

## 改造目标

请修改 `lark-whiteboard` 相关 Skill 文档与引用内容，使其满足以下要求：

1. 新增一个明确原则：
   - 对于流程图、泳道图、架构图、组织图、网络拓扑图、用例图、时序图，以及任何强调“节点可继续编辑、连线必须跟随节点移动”的任务，必须优先使用 `raw query -> raw modify -> raw update`。
   - 这类任务中，业务关系线必须使用原生 `connector`，不能退化成几何线条、路径线、图片或其他不可验证的表示。

2. 新增“业务关系连线”的定义：
   - 只要线表达的是步骤、角色、模块、服务、依赖、调用、回流、归属、设备连通、参与者关系等节点与节点之间的关系，这条线就是业务关系连线。
   - 业务关系连线的 `connector.from` / `connector.to` 必须引用节点 id。
   - 禁止把业务关系连线的端点写成 `{ x, y }` 坐标。

3. 限制坐标端点的适用范围：
   - 坐标端点仅允许用于坐标轴、尺寸标注、注释箭头、装饰线、无附着对象的辅助线。
   - 明确写入：这些线不属于主业务关系，不得替代节点间真实连接。

4. 限制 `waypoints` 的用途：
   - `waypoints` 只能控制路径，不得替代节点绑定。
   - 即使使用 `waypoints`，业务关系线两端也必须仍然是节点 id。

5. 修改 Skill 的“修改 Workflow”与“复杂图表 Workflow”：
   - 对“已有复杂画板修改/重排/补线/修线/补节点后重连”这类任务，不要默认把模型路由到外部图语言或重画路径。
   - 增加一个 raw-first 分支：先查询 raw，再在原生节点结构上做增删改和重连。
   - 只有当用户明确要求“整板重做且不要求后续编辑吸附能力”时，才允许进入其他表达路径。

6. 补充结构级校验规则：
   - 写回前检查所有业务 connector 的 `from` / `to` 是否都是字符串节点 id。
   - 检查这些 id 是否在当前 `nodes` 中存在。
   - 检查 connector 是否仍为原生 connector，而不是被做成其他节点类型或不可编辑图元。
   - 检查是否存在“线贴着边但并未绑定节点”的伪连接。
   - 校验失败则不得写回。

7. 补充回写后验证规则：
   - 要求再次读取 raw，核对关键业务关系是否仍然是 `node id -> node id`。
   - 在 Skill 中明确“拖动语义自检”：如果节点移动时连线理论上不会跟随，说明结构不合格。

8. 更新 references/connectors.md、schema.md、相关 scene 文档：
   - 把“业务关系连线必须是节点 id 到节点 id”写进去。
   - 把“坐标端点只用于注释/装饰/坐标轴”写进去。
   - 把“waypoints 不替代绑定”写进去。
   - 对容易误用的示例增加反例说明。

9. 修改文档措辞：
   - 不要把“视觉对齐”“看起来连上了”当成成功标准。
   - 成功标准必须是：结构真实绑定、锚点方向合理、拖动节点时连线应随之保持连接。

## 交付要求

请直接修改 Skill 文件，不要只给建议。交付时请说明：

- 改了哪些文件
- 新增了哪些硬约束
- 哪些旧路由或旧表述被收紧了
- 如何保证以后生成的是“真实吸附连接”，而不是“视觉对齐”

## 额外限制

- 不要输出“只是 prompt 问题”之类的归因判断。
- 不要写任何把本任务重新引向外部图语言的建议。
- 不要用模糊措辞，例如“尽量”“优先考虑但可灵活处理”。
- 这里要的是强约束、可校验、可拒收。
